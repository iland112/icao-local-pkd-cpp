## 문서 1: 전자여권시스템 현황 및 계획서 (6페이지)

### 1. 현행 시스템 운영 현황 (p.1 아키텍처 다이어그램)

**전체 흐름:**
```
여권 → 여권판독기 → MRZ/OCR → BAC/PACE → 전자칩 → ICRM 화면 표시
                                                    ↑
                                              PKI Toolkit (PA 검증)
                                                    ↑
                                         Local PKD (CSCA/DSC/CRL 캐시)
                                                    ↑
                                    PKD Manager → ICAO PKD (LDAP)
```

**현행 검증 프로세스:**
- BAC, CA, AA 인증 수행 후 전자칩으로부터 DG, SOD 등 수신
- PKI Toolkit이 PA 인증 (전자서명 검증, 인증서 체인 검증, DG 해시 검증) 수행
- Local PKD 또는 단말 캐시로부터 CSCA·DS 인증서·CRL 활용
- SOD 내 DS 인증서 포함 시 검증 무선 이용
- ICRM 화면에 "국가별 특성" 적용, MRZ·DG1 비교, 결과코드 생성 → EPS_ERR_CD (ICRM 그룹코드 184) → **"성공" 또는 "실패"** 최종 표출

### 2. 인증서 수집 경로 (p.2 테이블)

| 수집 경로 | 정보 종류 | 수집 형태 | 비고 |
|--------|--------|--------|-----|
| **외교적 경로** | CSCAC, DSC, JRL | 공문, 저장매체 수신 | |
| **ICAO PKD LDAP** | DSC, CRL, ML, DL, Non-Conformant | LDAP 연결 | CSCAC는 ML 형태로만 공개, ICAO M.L.S. 동기 |
| **ICAO Web Download** | ML | 파일(jdf 파일 다운로드) | Non-Conformant는 2021년 이전 정보로 수집, pkddownloadsg.icao.int |
| **ICAO** | ICAO ML | www.icao.int/Security/FAL/PKD/Pages/icao-master-list.aspx | |

**현재 문제:** 인증서 수집의 형태는 "우리부 이동" (= 법무부 자체 이관) 상태

### 3. 현행 기술 인증 체계 (p.2)

- **판독기 인증:** LDS 1.7 기준, BAC/CA/AA 수행 후 전자칩으로부터 DG, SOD 수신
  - ⚠ **LDS 1.8 및 PACE 미지원**
- **PKI Toolkit 인증:** PA 인증을 순차 수행, Local PKD 또는 단말 캐시에서 CSCA·DS·CRL 활용
- **ICRM 인증:** 국가별 특성 적용, MRZ·DG1 비교, 결과코드 생성

### 4. 전자여권시스템 문제와 개선 방안 (p.3~5) — 10개 문제점

| # | 문제점 | 상세 내용 |
|---|------|--------|
| 1 | **변경된 기술 표준 미지원** | ICAO가 LDS 1.8, PACE 등을 신규 도입, 2028년부터 각국의 의무 적용 권고 중. 현 시스템은 미지원하므로 PACE 단독 적용 전자칩 판독 불가. 서명 알고리즘 등 신규 기술 표준 식별 시스템 적용 필요 |
| 2 | **비규격 여권 호환성 부족** | 전자여권 칩의 표준 미준수 등 발급상 결함 발견은 일반적 현상("별첨 3" 참조). 유연한 방식으로 전자칩 검증이 가능함에도 시스템 운영에 미고려. 전자칩 검증 코드에 인증서(X.509), SOD(CMS) 등 '포맷'에 관한 검증 완화하거나 등 보완 로직 추가 필요. 과거(2013~2014년) 감식과 유형 분류를 통한 전자여권 오류 분석 및 소프트포럼 합 시스템 개선 시의 축적("별첨 4" 참조) |
| 3 | **인증시스템 노후화** | 2009년 도입된 서버 등 장비의 내용연수(6년) 초과 및 소프트웨어 지원 만료, PKD Manager 기능 운영 제한 및 ICAO PKD – Local PKD 연계 미작동 → **전자여권시스템 인프라 등 제반 환경 재구축 필요** |
| 4 | **ML 미활용** | ICAO PKD의 다운로드 웹페이지의 각국 및 ICAO ML은 CSCA 인증서의 수집 경로로서 보조적 활용 가능하나 현재까지 활용 부재. → ICAO ML 활용하여 CSCA 인증서 확보(LDAP 연결 및 파일 다운로드), UN CSCA 인증서 및 ML Signer 인증서를 활용한 ICAO ML 검증 자동화 필요 |
| 5 | **DL, Non-Conformant 미활용** | 각국 DL과 Non-Conformant의 표준 미준수 사항 관리 및 사용자 조회 기능 신설 필요 |
| 6 | **검증 결과 인식성 부족** | 전자여권시스템은 위·변조 여부를 식별할 목적으로 구축되었으나, 전자칩 검증 결과 "실패"가 위·변조만을 의미하는 것이 아니므로 의미 전달 불명확. "성공"과 "실패"로 양면적으로 구분할 수 없는 속성 → 다양한 결과에 대해 전자여권의 **'신뢰도'를 최종 결과로 도출**하고 최종 결과코드를 재정비 필요 |
| 7 | **전자칩 분석도구 부재** | 위·변조 의심 시 상세 분석 가능한 소프트웨어 부재, 전자적 검증 중 특이사항 발견 시 물리적 육안 식별에 의존 → 기존 전자여권 경필검증 기능 강화(GRT 등 기능 운용) 필요 |
| 8 | **국가별 특성판단 효과 부족** | 전자칩에 적용된 인증의 종류 판별, 전자칩 검증의 강제 성공 처리 등의 실무상 사용 부적합. 현실적 운영 환경을 고려하여 국가별 특성 관리 기능 폐지 |
| 9 | **모니터링 체계 부재** | 각국 전자여권 검증 시 발생하는 오류 중 빈도가 높은 유형을 사전 탐지하여 인증서 업데이트, 검증 조치 등의 조치 대상으로 모니터링 기능 부재 → **전자여권시스템 모니터링 기능 신설** |
| 10 | **인증서 관리 제약** | 전자여권시스템 운영에서 인증서 관리는 매우 중요하나, PKD Manager 등 별도 관리 기능(솔루션)은 접근성이 낮고 관리 현황 등 확인이 어려움. → 기존 PKD Manager의 인증서 관리 기능을 ICRM에 신설 |
| (추가) | **서비스 범위 한정** | 현재 출입국심사에만 한정, 인천공항공사·재외공관 등 서비스 필요성 존재. → 입·출무인 심사 대상으로 우선 구축하고, 안정화 이후 서비스 범위 단계적 확장 추진 |

### 5. 사업 추진 방안 (p.5~6)

**사업 목적:** 전자여권시스템 전면 재구축 및 인증 체계 고도화

**사업 방향 (4개):**
1. 전자여권시스템을 변경된 기술 표준 등 바탕으로 재구축 현대화
2. 전자여권시스템 운영 관리성 및 전자칩 분석 능력 강화
3. 전자여권 검증을 위한 PKD 등 기반 체계 활용성 강화
4. 전자여권 검증 결과의 신뢰성 강화

**사업 과제 (3개 대분류):**
- 전자여권 판독기 인증 (재)구축: BAC/PACE, AA, EAC 인증 및 판독 시스템 재구축
- 전자여권 PKI Toolkit 인증 (재)구축: PA 인증 시스템 재구축
- 전자여권 검증 관련 ICRM 기능 (재)개발
  - 출입국심사 등 화면의 전자여권 검증 기능 재개발
  - 전자여권 정밀 검증 기능 재개발
  - 전자여권 분석 기능 개발
  - 전자여권 검증 오류 모니터링 기능 개발
  - Local PKD 관리 기능 개발
  - 각국 DL 및 Non-Conformant 조회 기능 개발
  - 관련 통계 기능 재개발
- 전자여권 PKI (재)구축: Local PKD 재구축, ICAO PKD – Local PKD 연계

| **Local PKD 관리 기능 개발** | ICRM을 통한 Local PKD 조회, 공개 키 인증서 관리. CSCA, CSCA Link, DS 인증서(cer, der, pem 등), CRL(crl), ML(ml), DL(dl) 서명 및 검증 별 업로드. ICAO Web 다운로드된 인증서 등 파서(ldif) 업로드. 관리 프로세스 및 인증서 검증 |

| 관련 통계 기능 재개발 | 전자여권의 인증 적용 현황, 검증 실적 및 현황 |
| **Local PKD 재구축** | 전자여권 검증에 필요한 각국 모든 정보(CSCA·DS 인증서, CRL 등) 수집. ICAO PKD 인계를 위한 절차 수행 필요 |
| **ICAO PKD – Local PKD 연계** | 기존 서비스 관계를 고려하여 ICAO PKD 인계 요청 |

## 문서 2: 사업계획서 Part 1 (10페이지) — 22개 개발 과제 상세

| # | 과제명 | 핵심 내용 |
|---|------|--------|
| 1 | **여권 판독 공통 프로그램 개발** | 여권판독기로 여권 감지 → MRZ 영역 OCR → VIZ → RFID 칩 접근 → BAC/PACE/EAC-CA 등 인증 → LDS 1.8, ISO/IEC 30794 등 적용 |
| 2 | **전자여권 인증 공통 프로그램 개발** | BAC, PACE, AA, EAC-CA, EAC-TA, PA 인증 수행. Local PKD를 이용한 인증서 체인 검증 수행. 비표준 전자여권의 결함처리 포함. 전자여권 검증 결과에 대해 각 단계별·국가별 모든 에디정보를 인터페이스에 저장 |
| 3 | **Local PKD 구축** | 법무부 전자여권판독시스템의 전자여권 인증 운영에 이용되는 국가기관별 공개키 인증서 등 보관소 구축. ICAO PKD로부터 ICAO ML 및 국가기관별 ML, DS 인증서, CRL, DL 등을 LDAP 및 Web 경로로 추기적으로 자동 수신. 자동 수신 외 경로 확보된 공개키 인증서 등은 Local PKD 관리 프로그램을 통하여 자체 데이터베이스에 수집 저장 |
| 4 | **Local PKD 관리 프로그램 개발** | ML, CSCA 인증서, DS 인증서, CRL, DL 및 기타 인증서 포맷 관련(cer, con, pem, jdf, ml, bin 등). 사서치 조회 조건 상세 입력과 유관 목록 조회 및 전자증명별 상세. 결함정보 인증서 등 결합 국가기관별, 유형별, 버전별 등 기준으로 분류하여 정밀히 조회. 공개키 인증서 등 수동 업로드. 국가기관별 Trust Anchor를 분류 및 관리하고 자동 검증 활용. UN, 참부기관 등 국가기관 최상위 신뢰 CSCA 인증서, ML Signer 인증서, DL Signer 인증서 등을 분류 축적. CSCA 및 CSCA Link 인증서, DS 인증서, ML, DL 등 업로드 시 Trust Anchor로부터 전자서명 검증 등 동작 |
| 5 | **전자여권 운영환경 관리 프로그램 개발** | 전자여권의 판독 과정에서 수행하는 각 인증 적용 여부를 각 기관별, 단말별, IP주소별 및 실시유형(단말/무인원) 설정. EAC-TA 인증을 위한 CVCA 인증서, DV 인증서 관리 및 IS 인증서 분별 등 인증서 관리 기능 설치. 여권 판독 유형별 메시지 및 조치 관리기능 개발 |

## 핵심 시사점 — FASTpass® PKD 제안서 반영 포인트

### 1. ICRM이 핵심 연동 대상
현 시스템의 모든 검증 결과는 ICRM(Immigration Control & Recognition Module) 화면을 통해 표출. 5개 업무 영역(입·출국심사, 자동심사/SES, 체류관리, 사법처리, 국적심사)에서 사용. FASTpass® PKD의 REST API가 ICRM 연동 인터페이스 역할을 수행해야 함.

### 2. "신뢰도" 개념 도입 요구
기존 "성공/실패" 이분법에서 다단계 신뢰도로 전환 필요. 법무부 문서에서 명시적으로 "검증 결과를 기존 '성공'과 '실패'에서 '신뢰도'로 전환"을 요구. FASTpass® PKD의 Trust Chain 검증 + ICAO 준수 레벨(COMPLIANT/WARNING/NON_CONFORMANT) + AI 포렌식 위험 점수(0~100)가 이를 충족.

### 3. 파일 형식 지원 범위
cer, der, pem, con, jdf, ml, bin, ldif 등 다양한 형식 요구. FASTpass® PKD는 이미 PEM/DER/P7B/LDIF/ML/DL/CRL 지원. con, jdf, bin 형식 추가 지원 검토 필요.

### 4. 5개 업무 영역 대응
- 입·출국심사 (과제 #15)
- 자동심사/SES (과제 #17, #18)
- 체류관리 (과제 #19)
- 사법처리 (과제 #20)
- 국적심사 (과제 #21)

### 5. DL/Non-Conformant 관리 기능 필수
별도 과제(#11)로 명시. FASTpass® PKD의 DSC_NC 보고서 및 DL 파싱 기능이 직접 대응.

### 6. 모니터링 에이전트
실시간/비실시간 알림 및 비정상 패턴 요약 표시 요구 (과제 #13). FASTpass® PKD의 SSE 기반 NotificationManager + Monitoring Service가 대응.

### 7. 사업 규모
- **총 사업비: 약 27.3억원** (SW개발 6.1억 + HW구매 7.5억 + SW구매 11.7억 + 상용SW 1.2억 + 감리 1.5억)
- **투입공수: 21 M/M** (PM 1명×6월, 아키텍트 1명×4월, 개발자 3명×6~8월)
- **기존 상용SW**: PKD 중계 에이전트 2.2억, PKD Manager 3.3억, ICAO PKI Toolkit 0.66억, LDAP 3.3억, 오라클 2.2억
- FASTpass® PKD는 PKD Manager + PKD 중계 에이전트 + LDAP + PA 검증 기능을 하나의 솔루션으로 통합 → **상용SW 비용 대폭 절감 가능**
